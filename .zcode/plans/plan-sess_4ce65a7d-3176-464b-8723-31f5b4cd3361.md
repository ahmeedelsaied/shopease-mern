# Sprint 6B — Order Tracking + Better Checkout

## Decisions locked from clarifications
1. **OrderTimeline → extend in place** (keep `{ status, timestamps, updatedAt, className }` prop signature so all 3 consumers keep working). Add the 2 new steps + optional per-step note (additive).
2. **Backend → add full `statusHistory`** subdoc, seed on create, append-on-PATCH-never-overwrite, expose `GET /api/orders/:id/tracking`, centralize the enum in a constants file (kills current duplication).
3. **Checkout → lift step state into the existing single page**, keep one `<form>`, conditionally render Shipping/Payment/Review sections, wire `CheckoutSteps current={step}`.
4. **Confetti → pure CSS keyframes** in `index.css`, mounted once on `OrderSuccess`. No npm dependency.
5. **Admin note support**: PATCH `/status` accepts an optional `note` surfaced in the timeline.

## Rule compatibility
- No existing page is rewritten. `OrderTimeline`/`CheckoutSteps` are *extended* in place with backward-compatible props. `Orders.jsx`/`AdminOrders.jsx`/`OrderDetails.jsx` keep working (they already pass `timestamps={order.statusHistory}` and `updatedAt`).
- Routing, auth, wishlist, reviews, product APIs untouched.
- Admin authorization reuses the existing `admin` middleware on `adminRoutes`.
- No new npm dependency (all confetti/steps CSS-only).

## PART 1 — Backend: order status timeline

### New file: `backend/constants/orderStatus.js`
- `ORDER_STATUSES` array: `pending, confirmed, processing, shipped, out_for_delivery, delivered, cancelled`.
- `TERMINAL_STATUSES = ['delivered', 'cancelled']`.
- Exports the canonical list; imported by both `models/Order.js` and `adminController.js` to kill the duplicated hardcoded arrays (rule: don't duplicate knowledge).

### Modify: `backend/models/Order.js`
- `import { ORDER_STATUSES } from '../constants/orderStatus.js'`.
- Change `status.enum` → `ORDER_STATUSES` (default `'pending'` unchanged).
- Add `statusHistory: [{ status: String(enum ORDER_STATUSES), timestamp: { type: Date, default: Date.now }, note: String, changedBy: { type: ObjectId, ref 'User' } }]` with `{ _id: false }` (or keep `_id` — choose `_id: false` to match `orderItemSchema` style). No `required` on sub-fields beyond `status`.
- **No migration**: existing orders keep `status='pending'`; `statusHistory` defaults to `[]` (means the timeline shows the current state via `updatedAt` fallback, exactly as today).

### Modify: `backend/controllers/orderController.js`
- `createOrder`: after `Order.create(...)`, seed history by setting `statusHistory: [{ status: 'pending', changedBy: req.user._id }]` in the create payload (one atomic step; no second save).
- No change to `getMyOrders` / `getOrderById` (they already return the full order document including the new `statusHistory` array).

### New file: `backend/controllers/trackingController.js`
- `getOrderTracking` (wrapped in `asyncHandler`): `Order.findById(req.params.id).select('orderNumber status statusHistory createdAt updatedAt')`. Ownership check identical to `getOrderById` (`order.user.toString() !== req.user._id.toString()` → 403). 404 if missing. Returns `{ success, data: { orderNumber, status, history: [...], createdAt } }`.
- This is a dedicated lightweight read endpoint the OrderDetails page can hit for the timeline (also keeps status reads decoupled from the full order fetch).

### Modify: `backend/routes/orderRoutes.js`
- `router.get('/:id/tracking', protect, getOrderTracking)` — mounted **before** `/:id` is irrelevant (Express matches `/my` then `/:id/tracking` then `/:id`; order them so `/my` and `/:id/tracking` are declared before `/:id`).

### Modify: `backend/controllers/adminController.js`
- `updateOrderStatus`: import `ORDER_STATUSES` from the constants file (replaces the duplicated `['pending', ...]` array). Validate `status` against it. Accept optional `note` from `req.body`. Append to `order.statusHistory` (`order.statusHistory.push({ status, note, changedBy: req.user._id })`) **in addition to** setting `order.status = status` — never overwrites prior entries. Keep the `admin` middleware gate (route-level, unchanged).
- Route already exists: `PUT /api/admin/orders/:id/status`. No route change needed.

### Verify backend starts
- Run `node server.js` (or `npm start`/whatever the package script is) to confirm it boots and connects — no DB migration script run; rely on Mongoose's safe enum change.

## PART 3 — Frontend: Better Checkout (lift step state, one form)

### Modify: `frontend/src/pages/Checkout.jsx`
- Add `const [step, setStep] = useState('shipping')` (one of `shipping|payment|review`).
- Split the existing single `<form>` into three conditional sections (still inside one `<form>` element so the submit handler and `submittingRef` double-click guard stay untouched):
  - **Shipping**: existing address fields. "Continue to payment" button validates required fields; on success `setStep('payment')`.
  - **Payment**: existing `paymentMethod` (still Cash on Delivery — show it as a read-only selected card plus a "Pay on delivery" explainer; no real payment flow). "Continue to review" → `setStep('review')`. Back → `setStep('shipping')`.
  - **Review**: shows shipping recap + items recap + the existing "Place order" submit button. Back → `setStep('payment')`.
- `CheckoutSteps current={step}` (was locked to `'shipping'`).
- **Form validation**: a small `validateShipping(form)` helper returning a `{ [fieldName]: message }` map; render `<Input error={...}>` per field (the `Input` component already supports `error`). Disable "Continue" while invalid (show errors on blur/attempt). ZIP/phone light pattern checks.
- **Disable submit while processing**: already present via `loading`+`submittingRef`. Keep it; the loading button already swaps label. Add the `loading` state to the final review button too.
- **Sticky summary on desktop**: `Card variant="summary"` aside gets `lg:sticky lg:top-6` and `self-start` on the aside wrapper. Mobile: summary stays inline above/below (current flow).
- **Mobile-optimized summary**: a collapsible "Order summary" toggle on small screens (`lg:hidden`), expanded by default on `lg`.
- **Optimistic UI**: on submit the button optimistically shows "Placing order…" and disables before the await resolves (already true). Keep the existing toast on error (remove the *duplicate* inline `<p>` error + toast both firing — keep toast only as the single source, with one inline error line for accessibility).
- No new APIs, no routing change, `clearCart()` + `navigate('/order-success/:id')` unchanged.

### `CheckoutSteps.jsx` — **no change**. It already accepts `current` as a prop; lifting the state in `Checkout.jsx` makes the existing indicator actually drive.

## Risks / mitigations
- **OrderTimeline note rendering**: detect old vs new `timestamps[step.key]` shape (string vs object) defensively so `Orders.jsx`/`AdminOrders.jsx` keep working even if they still pass the old shape.
- **Mongoose enum widening**: safe for existing docs (no values are removed). No migration script.
- **Checkout rewrite risk**: minimal — keep the existing `<form>`, `submittingRef`, `clearCart`, `navigate`, and the order-summary aside; only wrap sections in `step` conditionals. Submit logic untouched.
- **Confetti leaks**: tie to a short timer + `prefers-reduced-motion`; no intervals kept after unmount.
